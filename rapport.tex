\section{Introduction et besoin identifié}
%% ==============================================================

\subsection{Contexte général}

Dans le domaine de l'administration système Linux, la sécurité des serveurs et postes de travail est une préoccupation permanente. Les attaques par élévation de privilèges, les mauvaises configurations et les binaires vulnérables constituent des vecteurs d'attaque fréquemment exploités par des acteurs malveillants.

La détection manuelle de ces risques est longue, sujette aux erreurs humaines et difficilement reproductible à grande échelle. L'automatisation de ces tâches d'audit représente donc un enjeu majeur pour toute organisation soucieuse de sa posture de sécurité.

\subsection{Besoin identifié}

\textbf{HARDEN} est un outil d'audit et de durcissement (\textit{hardening}) de sécurité Linux développé en Bash. Il automatise la détection des risques d'élévation de privilèges, des mauvaises configurations et des vulnérabilités courantes sur les systèmes Unix/Linux.

L'outil répond à un besoin réel des administrateurs système, des équipes de sécurité (Blue Team / Red Team) et des ingénieurs DevSecOps qui doivent effectuer des audits réguliers sur un ou plusieurs systèmes Linux.

\subsection{Objectifs du script}

\begin{itemize}[topsep=2pt, itemsep=2pt]
  \item Détecter automatiquement les binaires SUID vulnérables référencés dans GTFOBins
  \item Identifier les règles sudo dangereuses (NOPASSWD, wildcards)
  \item Analyser les tâches cron, les fichiers mal configurés et les clés SSH exposées
  \item Vérifier la configuration SSH, les capacités kernel et les permissions Docker
  \item Générer des logs horodatés avec des recommandations de correction (\cmd{Fix:})
  \item Supporter plusieurs modes d'exécution : normal, subshell, fork réel (C/\cmd{fork()}) et thread réel (C/pthreads)
  \item Fournir des fonctionnalités avancées : baseline SHA-256, auto-remédiation, export JSON, banner grabbing, analyse CVE kernel
\end{itemize}

\subsection{Public cible}

L'outil s'adresse principalement aux :
\begin{itemize}[topsep=2pt, itemsep=2pt]
  \item Administrateurs système Linux et ingénieurs DevSecOps
  \item Équipes de sécurité (Blue Team pour l'audit défensif, Red Team pour l'identification de failles)
  \item Auditeurs de sécurité souhaitant automatiser les contrôles de conformité
  \item Étudiants souhaitant comprendre et renforcer la posture de sécurité d'un système Linux
\end{itemize}

\subsection{Architecture générale}

HARDEN est structuré autour d'un script principal Bash (\fichier{harden.sh}) et de deux helpers C compilés dynamiquement dans un répertoire sécurisé :

\begin{itemize}[topsep=2pt, itemsep=2pt]
  \item \fichier{fork\_helper.c} --- Implémente le vrai appel système \cmd{fork()} pour le mode \opt{-f} (background réel).
  \item \fichier{thread\_helper.c} --- Utilise les POSIX threads (\cmd{pthread\_create/join}) pour le mode \opt{-t}.
\end{itemize}

L'architecture repose sur des mesures de sécurité internes robustes :
\begin{itemize}[topsep=2pt, itemsep=2pt]
  \item \textbf{Gestion des logs} : Utilisation de \cmd{flock} sur \fichier{history.log} pour éviter les race conditions en mode concurrent.
  \item \textbf{Isolation du Build} : Utilisation de \cmd{mktemp -d} pour \cmd{BUILD\_DIR} afin de prévenir les attaques par lien symbolique.
  \item \textbf{Nettoyage automatique} : Un \cmd{trap '...' EXIT} garantit la suppression des fichiers temporaires à la fin du script.
  \item \textbf{Codes d'erreur standardisés} : Gestion des erreurs via des codes spécifiques (100--105).
\end{itemize}

L'architecture des fonctions de scan est modulaire :

\begin{lstlisting}[style=bash]
./harden [options] <repertoire>
  -c  --> quick_scan()    # 7 checks essentiels
  -C  --> full_scan()     # 20 checks complets
  -s  --> ( full_scan )   # sous-shell isole
  -t  --> thread_helper "cmd1" "cmd2"  # Thread1=checks 1-11 / Thread2=12-20
  -f  --> fork_helper "$0 _real_fork_target" &  # fork reel
\end{lstlisting}

Cette architecture en deux couches (Bash + C) respecte la directive du cahier des charges sur l'utilisation de \textit{«~scripts complémentaires en langage C~»}.

\subsection{Syntaxe d'utilisation}

Le script HARDEN respecte la syntaxe standard des commandes Linux (directive §3.2.4) :

\begin{lstlisting}[style=bash]
./harden.sh [OPTIONS] <répertoire>

Le paramètre \texttt{<répertoire>} est \textbf{obligatoire} pour les modes de scan (\opt{-c}, \opt{-C}, \opt{-t}, \opt{-s}, \opt{-f}). Il représente le répertoire cible sur lequel les vérifications sont effectuées.

\subsection{Gestion des logs}

Toutes les sorties sont simultanément affichées dans le terminal (codes couleur ANSI) et enregistrées dans \fichier{history.log} après stripping des couleurs via \cmd{sed}. Le format respecte strictement la directive \S{}3.2.2 :

\begin{lstlisting}[style=bash]
yyyy-mm-dd-hh-mm-ss : username : INFOS/ERROR/WARNING/CRITICAL : message
\end{lstlisting}

\textbf{Note sur les types de messages :} Conformément à la directive \S{}3.2.2, le script utilise les types \texttt{INFOS} pour les sorties standards et \texttt{ERROR} pour les erreurs. Les niveaux \texttt{WARNING} et \texttt{CRITICAL} ont été ajoutés pour affiner la sévérité des détections de sécurité.

\textbf{Améliorations techniques du logging :}
\begin{itemize}
  \item \textbf{Concurrence sécurisée} : Utilisation de \cmd{flock} sur le fichier log pour garantir l'intégrité des données lors d'exécutions parallèles (Thread/Fork).
  \item \textbf{Traçabilité Fork} : Les messages du processus fils sont tagués \texttt{[FORK:PID=...]} pour une identification claire.
  \item \textbf{Compteurs persistants} : Les compteurs d'incidents sont toujours incrémentés, même en mode dry-run (\opt{-n}).
\end{itemize}

L'emplacement du fichier log s'adapte automatiquement selon les privilèges :

\begin{itemize}
  \item \textbf{root :} \fichier{/var/log/harden/history.log}
  \item \textbf{non-root :} \fichier{\textasciitilde/.harden/history.log}
  \item \textbf{option} \opt{-l DIR} \textbf{:} répertoire personnalisé spécifié par l'utilisateur
\end{itemize}

\begin{capturebox}{cap:logs1}

\subsection{Helpers C compilés à la volée}

Conformément à la directive §3.2.2 permettant des \textit{«~scripts complémentaires en langage C~»}, HARDEN utilise deux helpers C compilés automatiquement avec \cmd{gcc} :

\subsubsection{fork\_helper.c --- mode \opt{-f}}

Ce helper implémente un vrai appel système \cmd{fork()} (via la fonction POSIX \texttt{fork(2)}). Le processus parent lance le fils et continue son exécution, tandis que le fils s'exécute en arrière-plan (\cmd{disown}) avec une redirection silencieuse de ses sorties standards.\\

\begin{lstlisting}[style=c]
pid_t pid = fork();
if (pid == 0) {
    execl("/bin/sh", "sh", "-c", argv[1], NULL);
} else {
    int status;
    waitpid(pid, &status, 0);
}
\end{lstlisting}

\subsubsection{thread\_helper.c --- mode \opt{-t}}

Ce helper crée de vrais POSIX threads via \cmd{pthread\_create()} avec \cmd{libpthread}. Les deux threads reçoivent des \textbf{commandes distinctes} (argv[1] et argv[2]) pour un vrai découpage fonctionnel, puis \cmd{pthread\_join()} synchronise leur fin.

\begin{lstlisting}[style=c]
// thread_helper.c : 2 commandes distinctes
typedef struct { char *command; } thread_data_t;
int main(int argc, char *argv[]) {
    pthread_t threads[2];
    thread_data_t data[2];
    data[0].command = argv[1]; /* Thread 1 : checks 1-11 */
    data[1].command = argv[2]; /* Thread 2 : checks 12-20 */
    pthread_create(&threads[0], NULL, run_command, &data[0]);
    pthread_create(&threads[1], NULL, run_command, &data[1]);
    pthread_join(threads[0], NULL);
    pthread_join(threads[1], NULL);
    return 0;
}
\end{lstlisting}

\subsection{Sécurité interne du script}

Le script HARDEN v4.0 intègre des mécanismes d'auto-protection pour garantir son intégrité et sa robustesse :

\begin{itemize}
  \item \textbf{Protection contre les liens symboliques} : Le répertoire de compilation \cmd{BUILD\_DIR} est créé via \cmd{mktemp -d}, garantissant un nom imprévisible et des permissions restreintes dans \fichier{/tmp}.
  \item \textbf{Sûreté de l'auto-remédiation} : L'utilisation de \cmd{eval} a été proscrite. Les commandes sont analysées et filtrées via une whitelist stricte (\cmd{chmod}/\cmd{chown}).
  \item \textbf{Protection des binaires système} : Une liste d'exclusion (\cmd{PROTECTED\_BINS}) empêche toute modification accidentelle ou malveillante des binaires de sécurité vitaux (\cmd{sudo}, \cmd{passwd}, etc.).
  \item \textbf{Exécution non-interactive} : L'utilisation de \opt{sudo -n} dans les tests de privilèges évite tout blocage du script en attente d'une saisie utilisateur.
  \item \textbf{Verrouillage atomique} : Le recours à \cmd{flock} pour les écritures dans le journal prévient toute corruption du fichier \fichier{history.log} lors d'accès concurrents.
\end{itemize}

\subsection{Mode fork (\opt{-f})}

disown "$child_pid"          # Détachement du processus

TARGET="$dir1"
full_scan                    # Parent interactif
\end{lstlisting}

\section{Conclusion}
%% ==============================================================

\textbf{HARDEN v4.0} est un outil d'audit et de durcissement de sécurité Linux complet, développé en Bash avec des extensions en langage C. Il répond à un besoin réel des administrateurs système en automatisant la détection de \textbf{20 catégories de risques}, tout en respectant l'ensemble des directives du mini-projet.

Le script implémente avec succès les quatre modes d'exécution requis (normal, subshell, fork réel et thread réel POSIX), une gestion robuste des logs horodatés avec verrouillage via \cmd{flock}, ainsi qu'une gestion complète des erreurs avec six codes de retour spécifiques.

L'utilisation de helpers C compilés (\fichier{fork\_helper.c} et \fichier{thread\_helper.c}) représente une plus-value notable : les appels système \cmd{fork()} et \cmd{pthread\_create()} sont réellement invoqués, garantissant une concurrence réelle et efficace.
